# 选品工作流设计

日期：2026-06-14

## 1. 背景

运营需要在 WebApp 内提交选品候选商品，主管或第二位运营同事在飞书多维表格中完成初选，采购在待询价清单中补充供应商和报价，最终形成完整选品记录。项目已有登录用户、角色体系和飞书读写封装，`.env.example` 也已预留 `LARK_TABLE_SOURCING`。

飞书中已存在 `16_选品池`，原字段主要服务 AI 选品分析。为保留旧数据和兼容后续 AI 分析，本设计复用该表并扩展工作流字段，而不是新建重复表。

## 2. 目标

1. 在 WebApp 内提供选品表单，支持运营填写候选商品并提交到飞书。
2. 表单字段包含 `OEM码`、`品牌`、`商品链接`、`英文名称`、`中文名称`、`近90天销量`、`eBay平均售价`、`选品备注`。
3. 提交时自动写入当前登录用户名到 `登记人`，写入当前时间到 `登记时间`，并设置 `选品阶段` 为 `初选待处理`。
4. 主管或第二位运营同事在飞书初选视图中填写 `初选结果`、`最高购入价格`、`初选备注`，并将入选记录推进到 `已入选待询价`。
5. 采购在待询价视图中填写 `供应商`、`供应商报价`、`采购备注`，并按进度更新 `选品阶段`。
6. 保持飞书写入安全开关，只有 `LARK_WRITE_ENABLED=true` 时允许真实写入。

## 3. 非目标

1. 一期不在 WebApp 内实现主管初选和采购询价编辑界面；这两步先在飞书视图中完成。
2. 一期不自动抓取 eBay 销量或售价，表单只接收人工填写的数据。
3. 一期不自动生成 SKU、不写入 `01_SKU主数据`，选品完成后的 SKU 建档仍按现有流程处理。
4. 一期不改造现有 AI 选品分析字段，也不迁移 `16_选品池` 旧数据。

## 4. 飞书表结构

复用 `16_选品池` 作为选品工作流表。新增或确认存在以下字段：

| 字段 | 类型 | 写入方 | 用途 |
| --- | --- | --- | --- |
| `OEM码` | 文本 | WebApp | 候选商品 OEM 编码 |
| `品牌` | 文本 | WebApp | 候选商品品牌 |
| `商品链接` | URL 文本 | WebApp | eBay 或其他参考链接 |
| `英文名称` | 文本 | WebApp | 英文商品名称 |
| `中文名称` | 文本 | WebApp | 中文商品名称 |
| `近90天销量` | 数字 | WebApp | 人工登记销量参考 |
| `eBay平均售价` | USD 货币数字 | WebApp | 人工登记平均售价 |
| `选品备注` | 文本 | WebApp | 登记备注 |
| `登记人` | 文本 | WebApp | 当前登录用户名 |
| `登记时间` | 日期时间 | WebApp | 提交时间 |
| `选品阶段` | 单选 | WebApp / 飞书 | 工作流阶段 |
| `初选结果` | 单选 | 飞书 | 入选、未入选或待补充 |
| `最高购入价格` | CNY 货币数字 | 飞书 | 可接受最高采购价格 |
| `初选备注` | 文本 | 飞书 | 初选说明 |
| `初选人` | 文本 | 飞书 | 完成初选的人 |
| `初选时间` | 日期时间 | 飞书 | 初选时间 |
| `供应商` | 文本 | 飞书 | 采购询价供应商 |
| `供应商报价` | CNY 货币数字 | 飞书 | 供应商报价 |
| `采购备注` | 文本 | 飞书 | 采购备注 |
| `询价人` | 文本 | 飞书 | 完成询价的人 |
| `询价时间` | 日期时间 | 飞书 | 询价时间 |

`选品阶段` 选项为：

- `初选待处理`
- `已入选待询价`
- `询价中`
- `已完成`
- `未入选`

`初选结果` 选项为：

- `入选`
- `未入选`
- `待补充`

## 5. 飞书视图

在 `16_选品池` 内维护以下视图：

| 视图 | 筛选 |
| --- | --- |
| `全部选品` | 不筛选 |
| `初选待处理` | `选品阶段 = 初选待处理` |
| `待询价清单` | `选品阶段 = 已入选待询价` |
| `询价中` | `选品阶段 = 询价中` |
| `已完成` | `选品阶段 = 已完成` |
| `未入选` | `选品阶段 = 未入选` |

已有默认 `Grid View` 保留不删除，避免影响既有使用者。

## 6. WebApp 信息架构

一期将选品表单嵌入现有 `数据录入` 页面，新增一个 Tab：

| 字段 | 值 |
| --- | --- |
| 页面 | `/data-entry` |
| Tab | `选品登记` |
| 模块 | 继续沿用 `数据录入` 导航项 |
| 访问 | 登录用户均可提交；后续如需限制，可按角色收紧 |

这样可以复用当前数据录入页的表单风格、Toast 提示、飞书连接状态和写入安全逻辑，不需要新增导航入口。

## 7. WebApp 表单

表单分三组展示：

1. 商品身份：`OEM码`、`品牌`、`商品链接`
2. 商品名称：`英文名称`、`中文名称`
3. 市场数据与备注：`近90天销量`、`eBay平均售价`、`选品备注`

必填校验：

- `OEM码`
- `英文名称`
- `中文名称`

数字校验：

- `近90天销量` 转为整数，空值不写入。
- `eBay平均售价` 转为两位小数数字，空值不写入。

提交成功后清空表单，并提示“已提交选品记录”。提交失败时展示后端返回的飞书错误。

## 8. API 与数据流

新增或扩展现有通用写入链路：

1. `lark-server.ts` 将 `sourcing` 映射到 `LARK_TABLE_SOURCING`。
2. `/api/lark/save-record` 的 `TABLE_MAP` 接受 `sourcing`。
3. `DATE_FIELDS` 为 `sourcing` 配置 `登记时间`、`初选时间`、`询价时间`。
4. `DataEntryPage` 的 `SourcingForm` 调用 `/api/lark/save-record`，传入 `table: "sourcing"`。
5. 表单提交前调用 `/api/auth/me` 读取当前用户，后端或前端组装：
   - `登记人 = me.name`
   - `登记时间 = 当前时间`
   - `选品阶段 = 初选待处理`

后端仍会通过 `assertLarkWriteEnabled()` 阻止未开启写入的环境误写飞书。

## 9. 错误处理

1. 用户未登录或 `/api/auth/me` 返回空用户名时，表单提示重新登录。
2. `LARK_TABLE_SOURCING` 未配置时，后端返回配置错误，前端展示错误说明。
3. `LARK_WRITE_ENABLED` 未开启时，沿用现有“飞书写入已关闭”错误。
4. 飞书写入失败时，前端展示后端错误；表单内容保留，避免重复输入。
5. 近 90 天销量或价格输入非法时，在前端阻止提交。

## 10. 测试

1. 单元测试覆盖 sourcing 表映射，确保 `LARK_TABLE_SOURCING` 可被 `getLarkTableId("sourcing")` 读取。
2. API 层测试覆盖 `save-record` 接受 `table: "sourcing"`，并能规范化 `登记时间`。
3. 页面测试或轻量单元测试覆盖表单字段校验、数字转换和默认阶段写入。
4. 完成实现后运行 `npm test` 和 `npm run lint`。

截至 2026-06-14，`16_选品池` 已扩展为单表多视图工作流：

1. 已新增工作流字段。
2. 已新增 `全部选品`、`初选待处理`、`待询价清单`、`询价中`、`已完成`、`未入选` 视图。
