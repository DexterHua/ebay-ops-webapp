# Main SKU 负责人写入降级设计

## 背景

`main` 环境的 SKU 主数据表单从 `/api/auth/me` 取得当前登录人的姓名，并把姓名作为 `负责人` 提交到 `/api/lark/save-record`。本地环境可通过 `lark-cli` 按姓名解析飞书用户，但 Netlify 生产环境只有应用凭证；飞书的用户搜索接口只支持用户身份，因此线上解析逻辑只接受 `open_id`。当前会话和账号数据又不包含 `open_id`，所以生产保存必然报错并阻断整条 SKU 记录。

## 目标

- SKU 保存不再因负责人无法解析而整单失败。
- 负责人只能由服务端当前登录会话决定，不能信任前端提交值。
- 已配置姓名到 `open_id` 映射时，继续把负责人写入飞书人员字段。
- 未配置映射时，先保存其余 SKU 字段，并返回明确的非阻断提醒。
- 保持现有 `01_SKU主数据` 表、字段名和写入接口不变。

## 方案

新增可选环境变量 `LARK_USER_OPEN_IDS`，内容为登录姓名到飞书 `open_id` 的 JSON 对象，例如：

```json
{"登录姓名":"ou_example"}
```

SKU 保存流程调整为：

1. 前端不再读取 `/api/auth/me` 或提交 `负责人`，只构造 SKU 业务字段。
2. `/api/lark/save-record` 在处理 `skuMaster` 时调用 `requireSession()`，从已校验会话取得登录姓名。
3. 服务端读取 `LARK_USER_OPEN_IDS`，仅接受格式合法的 `ou_...` 值。
4. 映射存在时，将 `负责人` 写成飞书人员字段所需的 `[{ id: open_id }]`。
5. 映射缺失或无效时，删除 `负责人` 后继续创建记录，并在成功响应中返回提醒：SKU 已保存，但负责人未写入。

这样不需要在生产环境保存用户访问令牌，也不需要扩大通讯录权限；同时避免客户端冒用他人身份。

## 错误处理

- 未登录或会话失效：拒绝 SKU 保存，不允许匿名写入。
- `LARK_USER_OPEN_IDS` 不是合法 JSON：视为配置缺失，SKU 仍保存并返回负责人提醒。
- 映射值不是合法 `open_id`：视为配置缺失，不把无效人员值发送给飞书。
- 飞书记录创建失败：保持现有失败响应，不伪装成成功。
- 负责人缺失：只产生成功响应中的 warning，不影响 SKU 主记录。

## 测试

先增加会失败的路由回归测试，再实现最小代码：

- 线上 SKU 请求携带姓名时，服务端忽略客户端负责人，并使用当前会话对应的映射 `open_id`。
- 映射缺失时仍调用 `createLarkRecords`，且字段中没有 `负责人`，响应包含 warning。
- 会话失效时不写飞书。
- 前端 payload helper 不再要求或输出负责人。
